

## 정의

- 문제가 명시한 규칙대로 시간/단계를 따라가며 상태를 갱신하는 풀이 유형
- "최적해를 찾는" 문제가 아니라 "정해진 규칙을 재현"하는 문제

---

## 그리디 vs 시뮬레이션 판별

### 지문 시그널

- **그리디**: "최대", "최소", "가장 ~한", "~할 수 있는 방법 중" 등 최적화 어휘 등장
- **시뮬레이션**: "~합니다", "~됩니다" 같은 단정문으로 모든 행동이 강제됨. 최적화 어휘 없음

### 판별 한 줄

- "이걸 어떻게 **고를까**?" → 그리디 후보
- "이걸 어떻게 **따라갈까**?" → 시뮬레이션

### 그리디 판별 체크리스트 (4개 모두 yes여야 그리디)

1. 매 순간 선택지가 여러 개 있는가?
2. "이걸 고르면 최적"이라는 기준이 있는가?
3. 그 기준대로 골라도 전체 최적이 보장되는가 (탐욕 선택 속성)?
4. 한 번 고른 선택을 번복하지 않아도 되는가?

---

## 시뮬레이션 신호 3종

1. **시간/단계 개념이 명시적**
    
    - "시각이 되면", "끝낸 시각", "다음 단계"
    - 시간이 흐르는 문제는 그 시간을 따라가야 한다
2. **상태 전이 규칙이 if-then으로 나열됨**
    
    - "X 상태에서 Y 이벤트 → Z로 전이"
    - 선택의 문제가 아니라 규칙의 문제
3. **출력이 결과 순서/상태**
    
    - "끝낸 순서대로", "최종 상태"
    - 수치 최적화(최대/최소 개수)가 아님

## 시뮬레이션 + 자료구조 조합

|규칙|자료구조|
|---|---|
|후입선출 ("가장 최근에 ~한 것부터")|스택|
|선입선출 ("먼저 들어온 순서대로")|큐|
|특정 우선순위로 처리|우선순위 큐|
|격자 + 방향 이동|2D 배열|

## 시뮬레이션 풀이 핵심 구조

### 상태 변수 분리

- **현재 진행 중**: 단일 변수 (`cur`)
- **대기/멈춤**: 자료구조 (stack, queue 등)
- **미래 입력**: 인덱스 (`i`)
- **시간**: 시각 변수 (`time`)

이 셋을 **같은 자료구조에 섞어 넣지 않는다**. 상태가 다르면 변수도 분리.

### 시간 처리 두 방식

1. **이벤트 기반**: "다음 이벤트 시각으로 점프"
    - 빠르지만 빈 시간 처리 로직 필요
2. **단위 시간 기반**: "1분씩 흐르게"
    - 직관적, 빈 시간이 자연스럽게 처리됨
    - 시간 범위가 작을 때 유효

### 매 단계 처리 순서 (단위 시간 기반)

```
1. 지금 시각에 새 이벤트 있나? → 처리
2. 현재가 비었으면 대기 큐/스택에서 보충
3. 현재 1단계 진행
4. 현재가 끝났는지 체크 → 끝났으면 결과 기록 + 비우기
5. 시간 +1
```

## 패착 사례: 그리디 프레임으로 시뮬레이션 풀기

### 잘못된 사고 흐름

1. "정렬되어 있고 시간 다루니 그리디다"
2. "정렬 기준 + 선택 기준" 패턴 강제 적용
3. 한 번에 한 쌍씩 비교하며 누적 처리 시도
4. 상태 전이가 다단계인 케이스에서 깨짐

### 올바른 사고 흐름

1. 자연어로 작은 케이스 손으로 시뮬레이션
2. 필요한 변수와 자료구조 자연스럽게 도출
3. 그 다음에 패턴 분류 ("아, 이건 시뮬레이션 + 스택")
4. 분류 → 문제 (X), 문제 → 분류 (O)

### 핵심 교훈

- 알고리즘 문제를 **비즈니스 로직**으로 보면 풀리는 문제가 있다
- 작업 큐, 인터럽트 처리, 스케줄러 같은 실무 감각이 그대로 통한다
- 분류부터 정하면 그 틀에 끼워맞추다 막힌다

## 대표 문제: 과제 진행하기 (프로그래머스)

### 문제 요약

- 과제마다 시작 시각, 소요 시간 주어짐
- 시작 시각이 되면 진행 중인 과제 멈추고 새 과제 시작
- 끝나면 가장 최근에 멈춘 과제 재개
- 끝낸 순서대로 이름 반환

### 분류

- **시뮬레이션 + 스택**

### 핵심 인사이트

- "가장 최근에 멈춘 것부터" → 스택 (LIFO)
- 매 행동이 단정문으로 명시 → 선택 없음 → 그리디 아님
- 출력이 "끝낸 순서" → 재현이지 최적화 아님

### 구현 핵심

- `cur`: 현재 진행 중인 과제 (이름, 남은 시간) - 단일 변수
- `stack`: 멈춘 과제들 (이름, 남은 시간) - 별도 자료구조
- `time`: 흐르는 시각
- `i`: 다음 시작할 새 과제 인덱스 (또는 시작 시각 → 과제 map)

### 솔루션 (Kotlin)

```kotlin
fun calculate(plans: Array<Array<String>>): Array<String> {
    val answer = mutableListOf<String>()
    val stack = ArrayDeque<Pair<String, Long>>()
    val calculated = plans.map {
        it[0] to longArrayOf(parseTime(it[1]), it[2].toLong())
    }.sortedWith(compareBy { it.second[0] })

    val map = calculated.associate { it.second[0] to (it.first to it.second[1]) }
    var time = calculated.first().second[0]
    var remaining = calculated.size

    var curName: String? = null
    var curDuration = 0L

    while (remaining > 0) {
        // 1. 지금 시각에 새 과제 시작?
        val work = map[time]
        if (work != null) {
            if (curName != null) {
                stack.addLast(curName to curDuration)
            }
            curName = work.first
            curDuration = work.second
        }

        // 2. cur 없으면 stack에서 보충
        if (curName == null && stack.isNotEmpty()) {
            val (n, d) = stack.removeLast()
            curName = n
            curDuration = d
        }

        // 3. cur 1분 진행
        if (curName != null) {
            curDuration--
            if (curDuration == 0L) {
                answer.add(curName)
                remaining--
                curName = null
            }
        }

        time++
    }

    return answer.toTypedArray()
}
```

### 흔한 오답 패턴

1. **그리디로 분류** → 정렬 후 짝짝 처리하려 함
2. **`prevDuration += curDuration`** → cur가 깔끔히 끝난다고 가정 (cur도 멈출 수 있음을 놓침)
3. **stack top을 현재로 취급** → cur와 stack을 분리하지 않음
4. **남은 시간 갱신 안 함** → 멈출 때 원래 duration 그대로 push
5. **새 과제 들어올 때만 처리** → 그 사이 끝나는 멈춘 과제들 누락

### 모든 오답의 공통 원인

- "현재 진행 중"과 "멈춘 것"을 **별도 상태로 분리**하지 않음
- "흐르는 시간" 개념 부재

## 비슷한 시뮬레이션 문제 패턴

- **프린터 (큐)**: FIFO + 우선순위 재검사
- **다리를 지나는 트럭 (큐)**: 시간 흐름 + 적재량 제한
- **표 편집 (연결 리스트/스택)**: 명령 시퀀스 + 되돌리기
- **거리두기 확인하기 (격자)**: 2D + BFS/DFS
- **블록 이동하기 (격자)**: 2D + 상태 전이 + BFS

공통점: **규칙 강제**, **상태 갱신**, **최적화 아님**.
